How it works

The problems that came up building the Pips Solver, and how each was solved — the image parser first, then the energy-model solver, then getting it all to run in a browser.

← back to the solver

The challenge

A NYT Pips puzzle is an irregular board of cells grouped into coloured regions, each carrying a constraint (a target sum, <N/>N, “all equal” =, or “all different” ≠). You must tile every cell with a given multiset of dominoes so all constraints hold. Two distinct problems: read the puzzle out of a screenshot, and solve the constraint problem. The parser turned out to be the harder half. No machine learning and no network calls are used — just classical computer vision.

Part 1 · Parsing the screenshot

Finding the cell grid

Problem: cells sit on a square lattice, but the pixel size of a cell differs between screenshots, and the board is an arbitrary polyomino, not a full rectangle.

Solution: the cell pitch is recovered from the periodic dip in brightness at the thin gaps between cells — an autocorrelation of that signal gives the period regardless of image scale. The lattice phase is then locked by sliding the grid and maximising how much solid cell-tile it covers, which works on any polyomino shape.

Cell vs. hole vs. background

Problem: a lattice slot may be a coloured region cell, a neutral “free” cell, an empty hole, or just the beige board frame. The free-cell beige and the frame beige differ by only ~14 in RGB, so a colour threshold was hopelessly fragile (it kept dropping free cells or inventing phantom ones).

Solution: stop using a background colour at all. A real cell is a large, uniform tile that fills its slot; a non-cell is white, or a thin strip of frame. So a slot is a cell iff a single dominant colour covers most of its area — a structural test that treats free and region cells identically and needs no fragile colour reference.

Disconnected boards (the hard puzzles)

Problem: hard puzzles are several separate board pieces scattered across the canvas, and the pieces are not aligned to one shared lattice — a single global grid missed most cells.

Solution: split the board into connected components and fit a lattice phase to each piece independently (the pitch is global, the phase is per-piece). Slot sampling is restricted to the piece being processed so a tight gap between two pieces can’t bleed one into the other. Each piece gets its own coordinate band so dominoes can never wrongly bridge a gap.

Which cells form a region

Problem: clustering every cell colour globally merged genuinely separate regions that happened to share a hue (e.g. three different purple regions on one board).

Solution: a region is a connected component of equal-colour cells on the lattice. Same-coloured regions that are spatially apart stay distinct, which is exactly the game’s own rule.

Matching a constraint tag to its region

Problem: the little constraint markers are very saturated, while a region’s fill is a pale, desaturated tint — matching by raw colour failed, and several regions can share a tag colour. Nearest-cell matching swapped tags between neighbouring regions.

Solution: a region’s fill is its tag colour alpha-blended over beige, so the fill lies on the line segment between “beige” and “tag colour”. Tags are assigned to regions by an optimal (Hungarian) assignment on a cost that combines that colour-blend residual with spatial proximity: colour separates differently-hued neighbours, proximity separates regions that share a colour.

Reading the constraint text

Problem: the white glyph (a number, or =/≠/</>) is punched out of the solid tag, and small built-in OCR was unreliable on it. The = symbol’s bars even reach the tag edge, defeating naive flood-fill isolation.

Solution: recover the solid tag shape by morphologically closing its colour mask (this re-seals the punched text), and the glyph is then the bright pixels inside it. Split that into characters by column gaps and classify each against a small template library built once from real screenshots (tools/build_templates.py). New glyphs — >, the digit 0 from a “>0” tag, multi-digit numbers like 11/14 — were added by labelling them from new screenshots as they appeared.

Part 2 · Solving with an energy model

Problem: a brute-force tiling search is astronomically large, yet the answer must be exact — a “nearly right” board is wrong.

Solution: every region constraint contributes an energy that is zero exactly when satisfied and grows with the distance to satisfaction (see Constraint.violation). A valid solution is an assignment of total energy 0. The solver walks domino tilings depth-first, but for each region it keeps an admissible lower bound on the energy still reachable given the cells already filled (a partly-filled sum region can only land in a known window of totals; an “all-equal” / “all-different” region can only get worse). The branch is pruned the instant any region can no longer reach 0. That makes the search both exact and fast — milliseconds even for the 28-cell hard board.

Because a found solution is only as trustworthy as the parse, verify_solution independently re-checks the result from scratch: every cell covered exactly once, each pair of cells orthogonally adjacent, the domino multiset used exactly, and every constraint satisfied at energy 0. Nothing is reported as “solved” unless it passes that.

Part 3 · Running it in a browser

Problem: the parser/solver are Python (NumPy/SciPy/Pillow), but GitHub Pages only serves static files — no server to run Python on.

Solution: Pyodide runs CPython compiled to WebAssembly in the browser. The page loads NumPy, SciPy and Pillow, writes the unmodified pips package into the virtual filesystem, and calls it directly — so results are byte-identical to the command line. One sharp edge: GitHub Pages runs Jekyll, which silently drops files beginning with _ — including the glyph-template file. A .nojekyll marker disables that.

The solution is also drawn graphically as an SVG: region-tinted cells, with each domino as a rounded tile — blue and wide for horizontal, orange and tall for vertical — so the placement and orientation are obvious at a glance.
